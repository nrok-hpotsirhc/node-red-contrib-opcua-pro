Architektur- und Implementierungskonzept für einen Open-Source OPC UA Node in Node-REDEinleitung und tiefgreifende Analyse des OPC UA ProtokollsDie Open Platform Communications Unified Architecture (OPC UA) hat sich in den vergangenen Jahren als das dominierende und prägende Protokoll für den sicheren, plattformunabhängigen und zuverlässigen Datenaustausch im industriellen Umfeld etabliert. Um die Tragweite und die architektonischen Implikationen für die Entwicklung eines neuen Node-RED Nodes zu verstehen, ist eine detaillierte Betrachtung der Protokollmechaniken, der Historie und der zugrundeliegenden Paradigmen unerlässlich.In der Vergangenheit basierte der industrielle Datenaustausch auf dem sogenannten OPC "Classic"-Standard, welcher stark an die Microsoft-Technologien COM (Component Object Model) und DCOM (Distributed Component Object Model) gebunden war. Diese Abhängigkeit führte zu isolierten Dateninseln, da proprietäre Betriebssysteme und Netzwerkgrenzen eine nahtlose Integration erschwerten. Mit der Einführung von OPC UA wurde diese Limitierung aufgehoben. Die Architektur von OPC UA abstrahiert die eigentlichen Kommunikationsdienste vollständig von den transportierten Informationsmodellen. Diese Entkopplung ermöglicht eine semantische Interoperabilität, die cyber-physische Systeme über alle Ebenen der Automatisierungspyramide hinweg – vom einzelnen Sensor auf der Feldebene bis hin zu cloudbasierten Enterprise-Resource-Planning-Systemen (ERP) – miteinander verbindet. Diese Universalität macht OPC UA zu einem der wichtigsten Enabler für Industrie 4.0 und die digitale Transformation.Das Architekturparadigma: Client-Server und Publish/SubscribeDie fundamentale Architektur von OPC UA basiert auf einem robusten Client-Server-Paradigma, welches in neueren Spezifikationserweiterungen durch Publish/Subscribe (PubSub) Modelle ergänzt wurde. Im klassischen Client-Server-Modell obliegt es dem Client, eine sichere Sitzung (Session) mit dem Server auszuhandeln. Über diese Sitzung kann der Client synchrone oder asynchrone Dienste in Anspruch nehmen.Ein zentrales Element dieses Paradigmas ist das Konzept der "Subscriptions" (Abonnements) und "Monitored Items". Anstatt Daten durch kontinuierliches Polling abzufragen, was erhebliche Netzwerkbandbreite und Prozessorressourcen auf beiden Seiten binden würde, erlaubt OPC UA dem Client, bestimmte Datenpunkte zu abonnieren. Der Server überwacht diese Datenpunkte und überträgt neue Werte nur dann, wenn definierte Kriterien erfüllt sind. Hierbei kommt das Prinzip des "Report-by-Exception" zum Tragen, oft flankiert durch einen sogenannten Deadband-Filter. Dieser Filter stellt sicher, dass geringfügige Messrauschen oder Fluktuationen ignoriert werden und nur signifikante, absolute oder prozentuale Wertänderungen eine Datenübertragung auslösen.Darüber hinaus reicht das Paradigma weit über den reinen Datenaustausch hinaus. Clients haben die Möglichkeit, komplexe Programme oder Steuerungsroutinen auf dem Server über die sogenannten "Methods" direkt auszuführen. Diese Methoden können Eingabeparameter annehmen und Ausgabeparameter zurückgeben, was eine echte Remote Procedure Call (RPC) Architektur innerhalb des industriellen Netzwerks ermöglicht. Ebenso können Clients auf "Events" lauschen, um kritische Systemzustände, Alarme oder Statusänderungen in Echtzeit zu erfassen.Transportschichten und SicherheitsspezifikationenDie Flexibilität von OPC UA zeigt sich besonders in der Gestaltung der Transportschicht. Das Protokoll definiert nicht nur einen einzigen Transportweg, sondern bietet Optionen, die je nach Anwendungsfall skaliert werden können. Für hochperformante Echtzeitanwendungen auf der Maschinenebene wird primär der "OPC-binary over TCP" Transport genutzt. Dieser binäre Transport minimiert den Overhead und maximiert den Durchsatz. Für Szenarien, in denen Daten durch komplexe IT-Netzwerke oder Firewalls in die Cloud übertragen werden müssen, unterstützt OPC UA alternativ das JSON-Encoding über WebSockets.Die Sicherheit (Security) ist nicht als optionales Add-on konzipiert, sondern tief in den Kern des Protokolls eingebettet. Industrieanlagen sind zunehmend Ziel von Cyberangriffen, weshalb OPC UA eine mehrschichtige Sicherheitsarchitektur implementiert, die auf den Konzepten der Vertraulichkeit, Integrität und Authentifizierung beruht:SicherheitsmechanismusTechnische Umsetzung und ImplikationTransportverschlüsselungDie Kommunikation wird auf Transportebene verschlüsselt (z. B. durch Algorithmen wie Basic256Sha256). Dadurch wird das Abhören sensibler Maschinendaten durch Man-in-the-Middle-Angriffe unterbunden.NachrichtensignaturJedes ausgetauschte Datenpaket enthält einen kryptographischen Fingerabdruck. Der Empfänger kann somit zweifelsfrei verifizieren, dass die Nachricht während des Transports nicht manipuliert wurde.Sequenced PacketsDurch die strikte Vergabe von Abfolgenummern für jedes Paket wird das Risiko von Replay-Angriffen, bei denen Angreifer aufgezeichnete Befehle erneut senden, vollständig eliminiert.PKI AuthentifizierungJeder OPC UA Client und Server besitzt ein eindeutiges X.509-Zertifikat. Bevor eine Kommunikation stattfindet, müssen sich beide Parteien über eine Public-Key-Infrastruktur gegenseitig als vertrauenswürdig einstufen. Abgelehnte Zertifikate verbleiben in einem Quarantäne-Status.User Control & AuditingNeben der Applikationsauthentifizierung fordert das System eine Nutzerauthentifizierung (Benutzername/Passwort, Zertifikat oder JWT). Auditing-Funktionen erzeugen lückenlose Log-Trails aller ausgeführten Befehle, was für die Compliance in regulierten Industrien essenziell ist.Das Informationsmodell und der Address SpaceDie wahre Stärke von OPC UA liegt im objektorientierten Informationsmodell. Alle Entitäten, seien es physikalische Maschinen, Sensoren oder logische Datenstrukturen, werden als Nodes (Knoten) im sogenannten Adressraum (Address Space) des Servers abgebildet. OPC UA definiert acht fundamentale Node-Klassen: Object, Variable, Method, ObjectType, VariableType, ReferenceType, DataType und View.Jeder Node ist durch ein Set von Attributen charakterisiert. Das wichtigste Attribut ist die NodeId, die den Knoten global innerhalb des Servers eindeutig identifiziert. Die NodeId setzt sich aus einem Namensraum-Index (Namespace Index) und einem spezifischen Identifikator zusammen, welcher numerisch, als String, als GUID oder als Opaque formatiert sein kann (z. B. ns=3;s=Counter). Die Nodes sind über gerichtete Referenzen (References) miteinander verbunden, wodurch komplexe hierarchische Graphen entstehen. Beispielsweise kann ein Knoten vom Typ "Object" über eine "HasComponent" Referenz mit einem Knoten vom Typ "Variable" verbunden sein, welcher wiederum den aktuellen Temperaturwert abbildet.Moderne Speicherprogrammierbare Steuerungen (SPS) wie die Siemens S7-1500, Systeme von Rockwell Automation oder Omron nutzen intensiv User Defined Types (UDTs). Diese komplexen Datenstrukturen werden in OPC UA als sogenannte "Extension Objects" übermittelt. Ein hochwertiger OPC UA Client muss in der Lage sein, diese verschachtelten Strukturen nicht nur als rohe Byte-Arrays zu empfangen, sondern sie semantisch korrekt zu interpretieren und in nutzbare Formate umzuwandeln.Tiefgreifende Architekturanalyse bestehender Node-RED ImplementierungenNode-RED hat sich durch seinen Low-Code-Ansatz, die visuelle Programmierung und die ereignisgesteuerte Architektur als ideale Plattform für das Industrial Internet of Things etabliert. Die Herausforderung besteht darin, die hohe Komplexität des OPC UA Protokolls in einfach zu bedienende, aber dennoch performante Nodes zu abstrahieren. Um ein architektonisch fundiertes Konzept für einen neuen Open-Source-Node zu entwickeln, ist eine rigorose Analyse der bestehenden Repositories erforderlich. Alle relevanten Node-RED Implementierungen setzen auf der Basisbibliothek node-opcua auf, weshalb auch diese zunächst beleuchtet werden muss.Die Basis: Die "node-opcua" BibliothekDie Bibliothek node-opcua stellt die technische Grundlage für nahezu jede JavaScript-basierte OPC UA Implementierung dar. Entwickelt in TypeScript, implementiert sie den vollständigen OPC UA Stack vollständig asynchron und nutzt die Event-Loop-Architektur von Node.js optimal aus. Dies bedeutet, dass Netzwerkoperationen, kryptographische Berechnungen und das Parsing von Datenströmen den Haupt-Thread nicht blockieren. Die Bibliothek steht unter der MIT-Lizenz und zeichnet sich durch eine hohe Codequalität und eine stetige Weiterentwicklung aus. Für die Implementierung eines Clients bietet die Bibliothek ausgefeilte Mechanismen wie asynchrone Sequenzen (async.series), um den Lebenszyklus einer Verbindung von der Endpunktermittlung über die Zertifikatsvalidierung bis hin zur Sitzungserstellung deterministisch zu steuern. Auf der Serverseite ermöglicht node-opcua den Aufbau komplexer Informationsmodelle, die Unterstützung von Alarms & Conditions sowie die Integration von HistoryRead-Funktionalitäten.Analyse der Legacy-Implementierung: "node-red-contrib-opcua"Das Repository node-red-contrib-opcua und seine Derivate (wie das erweiterte node-red-contrib-iiot-opcua) haben die Verbreitung von OPC UA in der Node-RED Community maßgeblich geprägt. Initiiert und gepflegt von Entwicklern wie Mika Karaila und Klaus Landsdorf, boten diese Repositories erste praktikable Lösungen für den Datenzugriff.Dennoch zeigen diese Legacy-Nodes bei der Betrachtung durch die Linse eines Softwarearchitekten tiefgreifende konzeptionelle Schwächen, die ihren Einsatz in missionskritischen, hochskalierbaren industriellen Umgebungen stark einschränken. Die Analyse offenbart folgende Kernproblematiken:Architektonische SchwächeBeschreibung und Implikation für den BetriebDefizitäres Session-ManagementHistorisch gewachsen initiieren diese Nodes oftmals eine dedizierte Session pro konfiguriertem Lese- oder Schreibknoten im Flow. Selbst wenn rudimentäre Pooling-Ansätze vorhanden sind, führen diese bei komplexen Flows zu einer massiven Überflutung des Zielservers. Da speicherprogrammierbare Steuerungen (SPS) strenge Limits bezüglich der maximal erlaubten Sessions aufweisen, werden neue Verbindungsanfragen vom Server unweigerlich abgewiesen.Fehlendes Smart BatchingLese- und Schreiboperationen werden primär seriell abgearbeitet. Wenn in Node-RED hundert Sensordaten gleichzeitig ausgelesen werden sollen, generiert der Client hundert einzelne ReadRequest-Pakete. Dies multipliziert den Netzwerk-Overhead, erhöht die Latenz dramatisch und treibt die CPU-Auslastung auf der Steuerung in die Höhe. Ein Mechanismus zur Aggregation dieser Anfragen fehlt weitgehend.Instabilität des Reconnect-HandlingsDas sogenannte Auto-Healing bei Netzwerkausfällen ist fehleranfällig. Bricht die physische Verbindung ab, gehen Legacy-Clients oft in einen inkonsistenten Zustand über (z.B. "reconnect re-established"), reaktivieren die zuvor auf dem Server angelegten Subscriptions jedoch nicht korrekt. Die alten Sessions verbleiben als verwaiste Ressourcen auf der SPS, was zu einem Ressourcen-Leak führt. Um den Datenfluss wiederherzustellen, muss der Node-RED Flow oftmals manuell neu gestartet werden.Ergonomische Defizite im User InterfaceDie Adressierung zwingt den Nutzer, komplexe NodeId-Strings (wie ns=3;s=Global.Machine.State) fehlerfrei per Hand in Textfelder einzutippen oder über externe Tools wie den UaExpert zu ermitteln und zu kopieren. Eine visuelle, in Node-RED integrierte Lösung zur Exploration des Adressraums ist nicht nativ und benutzerfreundlich verankert.Verarbeitung komplexer DatenBeim Einlesen von User Defined Types (UDTs) scheitern die Nodes oft daran, die binären Extension Objects in nutzbare JavaScript-Objekte zu parsen. Der Nutzer wird mit undurchsichtigen Byte-Arrays zurückgelassen und muss die Dekodierung mühsam im Flow durch Funktion-Nodes abbilden.Einige dieser Probleme wurden durch Forks wie node-red-contrib-iiot-opcua adressiert, welches beispielsweise verbesserte Array-Unterstützung für Datentypen wie Float, Double oder Int64 sowie eine dynamische De-/Aktivierung von Nodes via Umgebungsvariablen (${OPCUA_ENABLE}) einführte. Dennoch bleibt das fundamentale Architekturproblem der unzureichenden Verbindungs- und Batch-Logik in diesen Open-Source-Derivaten bestehen.Analyse der kommerziellen Enterprise-Lösung: "@opcua/for-node-red"Ein starker Kontrast zu den Legacy-Nodes bildet das kommerziell vertriebene Modul @opcua/for-node-red, welches von Sterfive (den Maintainern von node-opcua) entwickelt wurde. Dieses Modul repräsentiert den State-of-the-Art der Architektur und definiert den Standard, an dem sich eine moderne Neuentwicklung messen lassen muss. Die Analyse dieses Repositories liefert essenzielle Blaupausen für den zu entwickelnden OSS-Node.Die Architektur dieses Moduls löst die Probleme der Vorgänger durch folgende Designentscheidungen:
Die Entwickler haben ein "Intelligent Session Sharing" implementiert. Unabhängig davon, wie viele Lese-, Schreib- oder Browse-Nodes im Node-RED Flow platziert werden, kommunizieren diese intern mit einer zentralisierten Verbindungsinstanz. Dies reduziert die Sessions auf der SPS auf ein absolutes Minimum.Flankierend dazu wurde ein "Smart Batching"-Algorithmus integriert. Dieser Algorithmus fängt asynchron eingehende Lese- und Schreibbefehle ab, puffert sie für wenige Millisekunden und fasst sie zu optimierten ReadMultiple- oder WriteMultiple-Requests zusammen. Große Anfragen werden bei Bedarf automatisch fragmentiert, um die maximalen Paketgrößen des Servers nicht zu überschreiten. Diese Maßnahme führt laut Dokumentation zu Performancegewinnen von bis zu 90 % bei umfangreichen Datentransfers.Ein weiteres herausragendes Merkmal ist das "Industrial-Grade Auto-Healing". Die Verbindungslogik basiert auf einer robusten Zustandsmaschine, die Netzwerkausfälle nicht nur erkennt, sondern über exponentielle Backoff-Strategien systematisch versucht, die Verbindung wiederherzustellen. Dabei werden bestehende Subscriptions durch die Reaktivierung der verlorenen Sitzungen nahtlos weitergeführt, ohne dass serverseitige Ressourcen erschöpft werden.Auch die User Experience wurde revolutioniert. Das Modul bietet einen "Visual Browser", der direkt in den Eigenschaften-Dialog des Node-RED Nodes eingebettet ist. Nutzer können den Baum des OPC UA Servers visuell durchstöbern, Ordner aufklappen und die gewünschten NodeIds mit einem simplen Klick übernehmen. Zusätzlich sorgt ein "Explore Node" dafür, dass komplette Teilbäume des Informationsmodells rekursiv ausgelesen und als perfekt strukturierte, verschachtelte JSON-Objekte in den Node-RED Payload übergeben werden. Die Handhabung von Zertifikaten wurde durch einen visuellen Certificate Manager vereinfacht, der den administrativen Overhead für den Aufbau von Vertrauensstellungen eliminiert. Schließlich basiert die gesamte Entwicklung auf einer Test-Driven Development (TDD) Methodik mit umfassenden Unit-Tests, was eine hohe Stabilität garantiert.Gap-Analyse und Zielsetzung für den neuen OSS-NodeDie Diskrepanz zwischen den kostenfreien, aber architektonisch veralteten Legacy-Nodes und der funktional überlegenen, jedoch kommerziell lizenzierten Enterprise-Lösung ist offensichtlich. Für akademische Einrichtungen, Start-ups und unabhängige Integratoren bildet diese Lücke eine erhebliche Barriere bei der Implementierung robuster IIoT-Lösungen. Der neu zu konzipierende Node muss exakt in diese Lücke stoßen. Er muss die Architekturmuster der Enterprise-Lösungen (Session Pooling, Smart Batching, visuelle Browser, stabiles Reconnect-Handling) adaptieren und unter einer permissiven Open-Source-Lizenz der Allgemeinheit zur Verfügung stellen. Dies demokratisiert den Zugang zu performanter industrieller Datenkommunikation und fördert die Verbreitung von Node-RED in professionellen Anwendungsszenarien.Lizenzrechtliche Strategie und Legal FrameworkEin zentrales Kriterium bei der Neuentwicklung von quelloffener Software für den industriellen Einsatz ist die Wahl einer rechtssicheren und angemessenen Open-Source-Lizenz. Anstatt die Verbindungslogik in den operativen Nodes (Read, Write, Browse) zu verankern, muss eine strikte Trennung von Verbindungsmanagement, Protokolllogik und Benutzeroberfläche erfolgen. Aus der Perspektive eines Softwarearchitekten stützt sich das Konzept auf drei zentrale Säulen.Säule 1: Zentrales Verbindungs- und Session-Pooling via Configuration NodeIn der Node-RED Architektur existiert das Konzept der "Configuration Nodes" (Config Nodes). Diese Nodes erscheinen nicht direkt auf dem Arbeitsblatt (Workspace), sondern laufen im Hintergrund und halten globale Zustände oder Konfigurationen, die von mehreren "regulären" Nodes im Flow referenziert und geteilt werden können. Ein klassisches Beispiel ist der MQTT-Broker-Config-Node, der eine einzige TCP-Verbindung für dutzende MQTT-In und MQTT-Out Nodes bereitstellt.Für den OPC UA Client wird ein dedizierter opcua-client-config Node entworfen. Dieser Node fungiert als exklusiver Connection Manager.
Das Herzstück dieses Config-Nodes ist eine Finite State Machine (Endlicher Automat), welche den Lebenszyklus der OPC UA Verbindung präzise orchestriert. Die definierten Zustände sind DISCONNECTED, CONNECTING, CONNECTED, SESSION_ACTIVE, CONNECTION_LOST und RECONNECTING.Wenn der Node-RED Flow gestartet wird (Deploy), evaluiert der Config-Node alle referenzierenden Worker-Nodes. Er baut exakt eine TCP-Verbindung und eine sichere OPC UA Session zum Zielserver auf. Jeder Zustandsübergang der State Machine löst einen internen Event-Emitter aus. Die abhängigen Worker-Nodes im Flow abonnieren diese Events und aktualisieren daraufhin ihr visuelles Status-Icon im Editor (z. B. ein roter Punkt für getrennt, ein blinkender gelber Punkt für Verbindungsaufbau, ein grüner Punkt für Session aktiv).Die Reconnect-Logik bei Netzwerkausfällen wird ausschließlich in diesem Config-Node abgewickelt. Kommt es zu einem Verbindungsabbruch, initiiert der Node eine exponentielle Backoff-Strategie (die Wartezeit zwischen Reconnect-Versuchen verdoppelt sich sukzessive bis zu einem Maximalwert). Wenn der Server wieder erreichbar ist, nutzt der Config-Node die Mechanismen von node-opcua, um die vorherige Session zu reaktivieren, anstatt sie blind neu zu erzeugen. Dies verhindert das in Legacy-Nodes beobachtete Leaking von Serverseitigen Ressourcen.Säule 2: Request Scheduler und Payload-Normalisierung (Smart Batching)Das zweite architektonische Kernstück adressiert die Netzwerk-Inneffizienz. Wenn ein Node-RED Anwender ein Trigger-Event nutzt, um zwanzig verschiedene Sensoren auszulesen, feuern zwanzig opcua-read Nodes nahezu gleichzeitig. Eine naive Architektur würde nun zwanzig asynchrone ReadRequest-Pakete über das Netzwerk schicken.Die neue Architektur implementiert einen Request Scheduler innerhalb des Config-Nodes. Jeder opcua-read oder opcua-write Node übergibt seinen Operationswunsch (z. B. "Lese NodeId X") an den Scheduler. Dieser speichert die Anfrage in einer internen Warteschlange (Micro-Task-Queue). Eine hochauflösende Timer-Funktion (z. B. mit einem Fenster von 5 bis 10 Millisekunden) verzögert die sofortige Ausführung. Sobald das Zeitfenster abgelaufen ist, analysiert der Scheduler die Warteschlange. Sind mehrere Leseanfragen vorhanden, aggregiert er diese zu einem einzigen, hochoptimierten OPC UA ReadMultipleRequest. Dieses Paket wird an den Server gesendet. Die Array-Antwort des Servers wird vom Scheduler wieder de-multiplexed. Anhand von intern generierten Promise-IDs oder Callbacks wird das jeweilige Resultat exakt an den Node-RED Node zurückgegeben, der die spezifische Anfrage gestellt hat. Dieser Prozess ist für den Anwender völlig transparent, senkt die Netzwerklast jedoch massiv.Zusätzlich übernimmt der Config-Node die Normalisierung der Daten. OPC UA liefert Daten als komplexe DataValue Objekte zurück, die neben dem eigentlichen Wert auch Timestamps (Source und Server) sowie einen Quality-Statuscode (z. B. Good, Bad, Uncertain) enthalten. Der Client wandelt diese Strukturen so um, dass der eigentliche Wert standardisiert im Node-RED msg.payload landet, während Metadaten in dedizierte Attribute (z. B. msg.opcua.timestamp) ausgelagert werden. Erkennt der Client beim Lesen ein Extension Object (UDT), durchläuft er eine rekursive Parsing-Schleife, um die binären Datenströme anhand der serverseitigen Datentyp-Definitionen in ein flaches, für Node-RED verdauliches JSON-Format zu konvertieren.Säule 3: Editor UX, Visueller Browser und PKI-ManagementDie User Experience entscheidet über den Erfolg eines Low-Code-Tools. Die Architektur erfordert eine nahtlose Integration in das Node-RED Backend und Frontend.Für den visuellen Tree-View Browser wird das Konzept der Node-RED Sidebar und der RED.treeList UI-Komponenten angewendet. Das Backend des Nodes (die JavaScript-Datei, die im Node.js-Prozess läuft) registriert eine dedizierte HTTP-Admin-Route, beispielsweise GET /opcua-admin/browse?nodeId=<Target>. Wenn der Anwender im Node-RED Editor den Konfigurationsdialog eines OPC UA Nodes öffnet, wird dort ein "Suchen"-Icon präsentiert. Ein Klick darauf öffnet einen modalen Dialog oder wechselt in einen Sidebar-Tab. Das Frontend sendet nun einen AJAX-Request an den Backend-Endpunkt. Das Backend öffnet eine flüchtige Browse-Session zum OPC UA Server, iteriert über die Kind-Knoten (BrowseDirection.Forward) und liefert die Hierarchie als JSON-Struktur an das Frontend zurück. Wenn Node-RED als OPC UA Server fungiert, wird die Low-Code-Plattform selbst zum Endpunkt, der Daten, die beispielsweise über MQTT, REST-APIs oder serielle Schnittstellen gesammelt wurden, für übergeordnete IT-Systeme (SCADA, MES, ERP) standardisiert bereitstellt.Aus architektonischer Sicht erfordert der Server eine völlig andere Herangehensweise. Das flüchtige, nachrichtenbasierte Modell der Node-RED Flows muss mit dem statischen, hierarchischen Konzept des OPC UA Address Space synchronisiert werden. Das Server-Konzept stützt sich auf folgende architektonische Prinzipien:Server-Lifecycle und Endpoint-ManagementAnalog zum Client wird ein zentraler opcua-server-config Node benötigt. Dieser initialisiert die Server-Klasse (OPCUAServer) von node-opcua. Er öffnet den Netzwerkport (standardmäßig 4840) und registriert die Endpunkte für die Verbindung. Der Server-Config-Node kümmert sich um den gesamten Lifecycle: Beim Start des Node-RED Flows (deploy) fährt der Server hoch, beim Stoppen führt er einen Graceful Shutdown durch und benachrichtigt alle verbundenen Clients. Der Config-Node beinhaltet zudem das PKI-Management, um Client-Zertifikate zu validieren und unerwünschte Verbindungsversuche auf Applikationsebene abzuwehren.Address Space Definition: Visuell und via XML-ImportDie größte architektonische Herausforderung ist die Modellierung des Informationsmodells. Externe Clients erwarten eine logische Ordner- und Variablenstruktur. Die Architektur sieht hierfür zwei parallel nutzbare Mechanismen vor:Dynamischer Node-Builder: Es werden spezielle UI-Nodes bereitgestellt (z. B. addFolder, addVariable), die vom Nutzer im Flow verschaltet werden. Beim Startprozess iteriert das Backend über diese Nodes und ruft sukzessive die node-opcua Methoden wie namespace.addFolder und namespace.addVariable auf, um den Address Space programmatisch zu konstruieren.NodeSet2.xml Importer: Für den professionellen Einsatz müssen standardisierte Companion Specifications (wie OPC UA for Machinery oder Euromap) unterstützt werden. Der Server-Node erhält eine Import-Funktion für standardisierte NodeSet2.xml-Dateien. Über die generateAddressSpace-Routinen von node-opcua parst der Server diese komplexen XML-Strukturen beim Startlauf und lädt hunderte von vordefinierten Datentypen, Objekttypen und Instanzen automatisch in den Speicher.Context-Mapping: Die bidirektionale DatenbrückeEin Server ohne dynamische Werte ist nutzlos. Wie gelangen die Sensordaten, die durch den Node-RED Flow fließen, in die statischen OPC UA Variablen? Die architektonische Lösung hierfür ist das Context-Mapping.
Node-RED bietet einen internen Speicher, den sogenannten "Context" (unterteilt in Node-, Flow- und Global-Context), um Daten persistieren zu können. Wenn ein Sensorwert im Flow ankommt, speichert ein Funktion-Node diesen Wert z. B. mittels flow.set("Sensor_A_Temp", msg.payload) ab.
Bei der Instanziierung der OPC UA Variablen bindet das Backend nun Getter- und Setter-Methoden an diese Variable. Wenn ein externer OPC UA Client den Wert der Variable anfragt, wird im Hintergrund die Getter-Funktion ausgelöst, welche den Wert via flow.get("Sensor_A_Temp") in Echtzeit aus dem Speicher von Node-RED ausliest und ausliefert. Schreibt ein externer Client einen Wert in den OPC UA Server, wird der Setter ausgelöst, welcher flow.set() beschreibt und optional eine msg im Node-RED Flow triggert. Diese bidirektionale Brücke entkoppelt die Taktung des Flows vollständig vom Polling-Intervall der Clients.Serverseitige MethodenaufrufeDas Konzept umfasst die Möglichkeit, dass Node-RED als intelligentes Backend für Method-Calls fungiert. Ein Node opcua-server-method wird bereitgestellt. Wird diese Methode im OPC UA Server von einem Client aufgerufen, generiert der Node ein Event. Dieses Event wird als msg-Objekt in den Flow gesendet, wobei die Input-Argumente des OPC UA Clients im msg.payload enthalten sind. Der Flow führt nun beliebige Logik aus (z. B. eine Datenbankabfrage oder das Senden einer E-Mail).
